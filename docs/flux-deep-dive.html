一、响应式编程基础

1.1 什么是响应式编程

想象你去餐厅点餐。有两种服务方式：

传统方式（阻塞）响应式方式（非阻塞）
你站在柜台前等厨师做完，
期间什么也干不了。
线程 = 你，阻塞等待 		你取个号坐下玩手机，
做好了服务员喊你取餐。
线程 = 服务员，来活了就处理
一个线程服务一个请求，请求多就开更多线程 少量线程处理海量请求，空闲时去做别的事

响应式编程的核心：把耗时操作（IO、数据库、LLM 调用）变成"发布者-订阅者"模式——你订阅一个数据流，数据到了自动通知你，不用傻等。

💡 一句话：阻塞是"等人送餐"，非阻塞是"取号等叫号"。

1.2 Publisher-Subscriber 模式

Reactor 的核心是 Reactive Streams 标准，定义了四个角色：

订阅关系链： Publisher (发布者) ← 数据源，比如 LLM 流式响应 │ subscribe(Subscriber) ▼ Subscriber (订阅者) ← 数据消费者 │ onNext(item) ← 来一个数据调一次 │ onComplete() ← 数据发完了 │ onError(e) ← 出错了 ▼ Subscription (订阅令牌) ← 控制流量 │ request(n) ← 我要 n 个数据 │ cancel() ← 我不想要了

关键：数据不是发布者硬塞的，而是订阅者通过 request(n) 拉取的——这就是背压的基础。

1.3 背压（Backpressure）

背压是响应式编程最核心的设计：消费者告诉生产者自己消化得有多快。

没有背压（传统队列）： 生产者 ──────[队列]──────→ 消费者 (拼命塞) 溢出！ (吃不下) 有背压（Reactive Streams）： 生产者 ←────request(1)───── 消费者 (等你吃完再给下一个)

Reactor 的 Flux.create 默认的背压策略是 BUFFER——消费者来不及消费时，元素在内部缓冲区排队。如果缓冲区满了，可以换策略：DROP（丢）、LATEST（只留最新的）、ERROR（抛异常）。

但注意：背压只解决消费者跟不上生产者的问题，跟本项目碰到的 Bug（跨线程导致 Netty 不 flush）是两回事。

💡 背压的直观理解：就像水管上装了个阀门，水流太快了阀门关小点，太慢了开大点。

二、Flux 核心概念

2.1 Flux vs Mono

Mono<T>Flux<T>
0 或 1 个元素 0 到 N 个元素
像 Java 的 Optional 像 Java 的 List
适用于：一次 HTTP 请求返回
一个 DTO		 适用于：LLM 流式响应
返回 N 个 token
Mono.just("hello") Flux.just("a", "b", "c")

本质上 Mono 是 Flux 的特例。Mono 能做的一切 Flux 都能做，反之不然。

2.2 Cold vs Hot Publisher

这是一个极其重要的概念，直接影响你对 blockLast() 行为的理解。

Cold Publisher（冷的）Hot Publisher（热的）
每个订阅者独立从头开始接收数据 所有订阅者共享同一个数据流
像 Netflix：每个人从头看 像 电视直播：来晚了就错过了
Flux.just()、Mono.just() Sinks.many()、Flux.interval()

关键理解：chatModel.stream() 返回的 Flux 是 cold 的。当你调用 .subscribe() 或 .blockLast() 时，它才开始真正的 HTTP 请求去拿 LLM 的数据。不订阅就不会有任何动作——这叫惰性求值。

⚠️ 这解释了为什么 Flux.create 内部调用 .blockLast() 会导致线程阻塞： blockLast() 会订阅这个 cold publisher 并阻塞当前线程直到数据全部到达。在 Spring WebFlux 的 EventLoop 线程上绝对不能这样做，所以之前才用 subscribeOn(boundedElastic) 把整个 lambda 扔到独立线程上执行。

2.3 线程模型：Schedulers

Reactor 里有几种内置的线程池：

Scheduler用途类比
Schedulers.boundedElastic() 阻塞操作
（JDBC、同步 API 调用）		 传菜员——负责需要等待的事
Schedulers.parallel() CPU 密集型计算 厨师——只管炒菜
Schedulers.single() 单线程任务 管理一个需要排队的资源
EventLoop（默认） Netty 的事件循环线程
（一切非阻塞 IO）		 大堂经理——不干重活，只负责调度

最重要的规则：EventLoop 线程绝不能阻塞。一旦阻塞，该线程上所有的 IO 处理（其他 HTTP 请求、WebSocket 连接、SSE 推送）全部卡住。这就是为什么之前的代码用 subscribeOn(boundedElastic) 把阻塞操作挪走。

2.4 非阻塞到底怎么实现的

2.4.1 传统的线程模型

// 传统 Servlet 容器（Tomcat）：每请求一线程
┌───────┐     ┌───────┐     ┌───────┐
│ 请求1 │ ──→ │ 线程1 │ ──→ │ 处理  │ ← 线程阻塞等待 JDBC
├───────┤     ├───────┤     ├───────┤
│ 请求2 │ ──→ │ 线程2 │ ──→ │ 处理  │
├───────┤     ├───────┤     ├───────┤
│ 请求3 │ ──→ │ 线程3 │ ──→ │ 处理  │
└───────┘     └───────┘     └───────┘
200 个并发 = 200 个线程 → 16GB 内存轻松吃掉

2.4.2 Netty 的 EventLoop 模型

┌───────┐
│ 请求1 │ ──┐
├───────┤   │  ┌───────────────┐
│ 请求2 │   ├─→│  EventLoop    │ ← 通常 CPU 核数个线程（比如 4 个）
├───────┤   │  │  (epoll/kqueue)│
│ 请求3 │ ──┘  └───────────────┘
└───────┘           │
                    ▼ 注册回调，不等待
        ┌───────────────────┐
        │  非阻塞 IO 操作    │ ← read/write 立即返回
        │  "数据到了叫我"    │ ← 通过操作系统的 epoll/kqueue
        └───────────────────┘
                    │
        数据到达 → 回调触发 → 处理

200 个并发 = 4 个线程，0 个线程在等 IO

核心秘密：Java 传统的 InputStream.read() 是阻塞的——没有数据时线程停住。Netty 使用操作系统提供的多路复用器（Linux 的 epoll、macOS 的 kqueue、Windows 的 IOCP），可以同时监控几千个 socket 连接，哪个有数据来了就处理哪个，没有数据就去做别的事。

epoll 的工作原理（极度简化）： ┌─ EventLoop 循环 ─────────────────────────┐ │ while (true) { │ │ events = epoll_wait(fds) ← 等最多 1ms │ │ for (ev : events) { │ │ if (ev 可读) handler.read() │ │ if (ev 可写) handler.write() │ │ } │ │ } │ └──────────────────────────────────────────┘ 一个线程同时管理几万个 socket。 没有数据时 epoll_wait 直接跳过，去干别的事。
💡 所以非阻塞的本质是：线程不再"挂起等待"数据，而是"这个连接没数据？我去看看其他连接"。
这要求所有 IO 操作都必须是非阻塞的——你不能在 EventLoop 线程上调用 Thread.sleep() 或同步的 InputStream.read()。一旦阻塞了某个 EventLoop 线程，它管理的几千个连接全跟着遭殃。

三、Flux 常用 API 范例

3.1 创建 Flux

Flux.just() — 从固定元素创建

Flux<String> stream = Flux.just("a", "b", "c");
stream.subscribe(System.out::println);
// 输出: a  b  c

Flux.fromIterable() — 从集合创建

List<String> list = List.of("x", "y", "z");
Flux<String> stream = Flux.fromIterable(list);

Flux.create() — 编程式发射（重要，跟 Bug 直接相关）

Flux<String> stream = Flux.create(sink -> {
    // 这个 lambda 被订阅时才会执行
    // sink 是线程安全的！可以从任何线程调用
    sink.next("a");
    sink.next("b");
    sink.complete();  // 必须手动结束
});

// 订阅后才真正执行上面的 lambda
stream.subscribe(System.out::println);

Flux.create 的特点：

  • sink.next() 可以从任何线程调用——包括 EventLoop 线程、boundedElastic 线程，甚至多个线程同时调
  • 内部有缓冲区，消费者跟不上时自动排队
  • 背压策略可通过 Flux.create(sink → {}, FluxSink.OverflowStrategy.BUFFER) 配置
  • Bug 关键：从不同线程调 sink.next() 虽然 sink 本身是线程安全的，但下游处理（比如 Netty 写入）的 flush 行为会受影响

Flux.defer() — 延迟创建（惰性求值）

// 每次订阅都重新执行 supplier
Flux<String> stream = Flux.defer(() -> {
    System.out.println("创建新的 Flux");
    return Flux.just("a", "b");
});

stream.subscribe();  // 输出: 创建新的 Flux
stream.subscribe();  // 输出: 创建新的 Flux（每次重新创建）

Flux.defer 在修复后的代码中用于 continuationFlux——确保工具执行逻辑在 LLM 流结束后才执行，而不是在构建 flux 链时就执行。

3.2 转换操作

map() — 一一映射（同步）

Flux.just("a", "b", "c")
    .map(s -> s.toUpperCase())
    .subscribe(System.out::println);
// A  B  C

map 是同步的，回调在发布者所在的线程上执行。适用于快速转换，不能做阻塞操作。

flatMap() — 一一映射到 Publisher（异步）

Flux.just("a", "b", "c")
    .flatMap(s -> {
        // 返回一个 Mono 或 Flux，它们可以异步执行
        return Mono.just(s.toUpperCase())
            .subscribeOn(Schedulers.boundedElastic());
    })
    .subscribe(System.out::println);

flatMap 与 map 的根本区别：

  • map 返回值直接是转换后的元素（同步）
  • flatMap 返回一个新的 Publisher（Mono 或 Flux），这个 Publisher 可以被调度到其他线程上执行
  • flatMap 内部可以改变元素数量：一个输入可以变成 0 个、1 个或多个输出

在修复后的代码中，我们用 flatMap 来实现 LLM token 的实时推送：

// 修复后：flatMap 在 EventLoop 线程上 emit thinking_token
chatModel.stream(...)
    .flatMap(chatResponse -> {
        AssistantMessage msg = ...;
        // 有文本 → 发射一个 thinking_token
        // 没文本（比如纯 toolCall 分片）→ 发射空
        if (msg.getText() != null && !msg.getText().isEmpty()) {
            return Mono.just(StreamChunk.thinkingToken(msg.getText(), ...));
        }
        return Mono.empty();
    })

concatMap() — 保证顺序的 flatMap

Flux.just(100, 10, 1)
    .concatMap(delay -> {
        // 保证输出顺序跟输入一致
        return Mono.just(delay).delayElement(Duration.ofMillis(delay));
    })
    .subscribe(System.out::println);
// 100 → 10 → 1（即使 1 的延迟最短，也要等前面的完成）

flatMap 不保证顺序（谁先完成谁先出），concatMap 保证顺序但牺牲了并发度。SSE 场景下，flatMap 更合适——我们不需要保证 token 顺序，LLM 本身返回的 token 已经是乱序的（文本和 toolCall 分片交替）。

3.3 组合操作

concatWith() — 串联两个 Flux

Flux.just("a", "b")
    .concatWith(Flux.just("c", "d"))
    .subscribe(System.out::println);
// a  b  c  d

// 修复后的核心用法：阶段1的 thinkingFlux 完成后接阶段2
return thinkingFlux.concatWith(continuationFlux);

concatWith 的行为：第一个 Flux 必须complete()后，才会订阅第二个 Flux。这完美匹配"先流式输出 LLM token，再执行工具"的场景。

merge() / mergeWith() — 合并两个流（交替输出）

Flux.interval(Duration.ofMillis(100)).map(i -> "A" + i)
    .mergeWith(Flux.interval(Duration.ofMillis(150)).map(i -> "B" + i))
    .take(6)
    .subscribe(System.out::println);
// A0  B0  A1  A2  B1  A3（交替输出）

3.4 副作用操作（doOnXxx）

Flux.just("a", "b", "c")
    .doOnNext(item -> System.out.println("收到: " + item))
    .doOnComplete(() -> System.out.println("完成"))
    .doOnError(e -> System.out.println("出错: " + e.getMessage()))
    .subscribe();
// 收到: a  收到: b  收到: c  完成

关键理解：doOnNext 不改变元素本身，它只是"经过时看一眼"。在修复后的代码中，我们用 doOnNext 做副作用（累计文本到 StringBuilder、收集 toolCalls），用 flatMap 做主逻辑（发射 thinking_token）：

chatModel.stream(...)
    .doOnNext(cr -> {
        // 副作用：累计状态
        thoughtBuffer.append(msg.getText());
        toolCalls.addAll(msg.getToolCalls());
    })
    .flatMap(cr -> {
        // 主逻辑：发射事件给下游
        return Mono.just(StreamChunk.thinkingToken(...));
    })

3.5 线程调度

subscribeOn() — 影响订阅的线程

Flux.<String>create(sink -> {
    // 这个 lambda 会在 BOUNDED_ELASTIC 线程上运行
    System.out.println("线程: " + Thread.currentThread().getName());
    sink.next("hello");
    sink.complete();
})
.subscribeOn(Schedulers.boundedElastic())
.subscribe();

// 输出: 线程: boundedElastic-1

subscribeOn 影响从哪里开始订阅。在 Bug 代码中，subscribeOn(boundedElastic) 把整个 Flux.create lambda 移到了 boundedElastic 线程上执行，这样内部的 blockLast() 才不会阻塞 EventLoop。

publishOn() — 影响后续操作执行的线程

Flux.just("a", "b", "c")
    .publishOn(Schedulers.boundedElastic())
    .map(s -> {
        // 这个 map 在 BOUNDED_ELASTIC 上执行
        System.out.println("线程: " + Thread.currentThread().getName());
        return s.toUpperCase();
    })
    .subscribe();

publishOn 割裂了执行链——之前的操作在原来的线程上，之后的操作在新线程上。修复后的代码中，continuationFlux 用 publishOn(boundedElastic) 确保同步的 tool.call() 在 boundedElastic 上执行，不污染 EventLoop。

subscribeOnpublishOn
影响范围 订阅动作（最上游） 下游操作（当前位置之后）
调用几次有效 只有第一次生效 每次调用都切线程
典型场景 把 Flux.create 扔到后台线程 把同步/阻塞操作扔到 boundedElastic

3.6 阻塞操作

block() / blockLast() — 不要轻易用！

// ❌ 危险！阻塞当前线程直到 Flux 完成
List<String> result = Flux.just("a", "b", "c")
    .blockLast();    // 返回最后一个元素 "c"

// 在非响应式代码中与响应式桥接时可以用，但绝不能
// 在 EventLoop 线程上调用！
⚠️ 本 Bug 的核心：blockLast() 在 Flux.create lambda 内部被调用。lambda 本身在 boundedElastic 上（因为有 subscribeOn），所以 blockLast 不阻塞 EventLoop。但 doOnNext 回调在 EventLoop 上，它里面调用 sink.next()——这就造成了跨线程 emission。

四、Bug 复盘：跨线程 sink.next() 导致 SSE 事件缓冲

4.1 问题现象

全流程流式改造完成后测试发现：

  • ✅ 第一个 thinking_token 前端能很快收到
  • ⏳ 然后陷入漫长的等待（LLM 还在生成，但前端没收到任何新事件）
  • 📦 最后一次性收到全部剩余事件（几十个 thinking_token + 工具调用 + 最终答案）

完全不是预期的"像 ChatGPT 一样逐字出现"的效果。

4.2 旧代码逐段分析

这是出问题的代码：

public Flux<StreamChunk> stream(String userInput, String conversationId) {
    return Flux.<StreamChunk>create(sink -> {       // ← [1]
        // ... 初始化 ...

        for (int i = 0; i < maxIter; i++) {    // ← [2] 指令式循环
            StringBuilder thoughtBuffer = new StringBuilder();
            List<ToolCall> toolCalls = new ArrayList<>();

            chatModel.stream(...)               // ← [3] 返回 cold Flux
                .doOnNext(chatResponse -> {      // ← [4] 回调在 EventLoop 线程
                    sink.next(thinkingToken);   // ← [5] ❌ 跨线程调用 sink.next()
                })
                .blockLast();                   // ← [6] 阻塞 boundedElastic

            // [7] blockLast 返回后，还在 boundedElastic 线程
            sink.next(toolCallStart);            // ← [8] 同一个线程调 sink.next()
            sink.next(toolCall);
        }
    })
    .subscribeOn(Schedulers.boundedElastic());  // ← [9]
}

逐段分析：

标记做了什么问题
[1]创建 Flux，sink 是线程安全的发射器✅ 没问题
[2]指令式 for 循环，在 lambda 内部✅ 没问题
[3]调用 LLM 流式 API，返回 cold Flux✅ 没问题
[4]doOnNext 回调注册到 inner Flux❌ 回调执行在 EventLoop 线程（Spring AI 的 HTTP 客户端是 Netty 的）
[5]在 EventLoop 上调用 sink.next()⚠️ sink 线程安全，但下游写入器（Netty）的行为受影响
[6]blockLast() 阻塞当前线程等 inner Flux 完成✅ 因为 [9] 把 lambda 移到了 boundedElastic，所以阻塞的是 boundedElastic 不是 EventLoop
[7]~[8]blockLast 返回后继续执行，此时在 boundedElastic 线程⚠️ 同一个线程调 sink.next()
[9]把整个 Flux.create lambda 移到 boundedElastic✅ 必须的，否则 blockLast 阻塞 EventLoop

4.3 线程交叉图解

执行过程中的线程流向：

时间轴 ↓　　　　boundedElastic 线程　　　　　　EventLoop 线程 　　　　　　　　　　　　　　　│　　　　　　　　　　　　　　　│ t0 Flux.create lambda 开始 │ ← subscribeOn 启动线程 │ t1 chatModel.stream() 订阅　│　　　　　　　　　　　　　│ t2 blockLast() 阻塞等待　　 │ ← 这里停住了　　　　　　│ 　　　　　　　　　　　　　　　│　　　　　　　　　　　　　│ t3 LLM 返回第一个 token │　　　　　　　　　　　　doOnNext → sink.next() ← 首次 flush ✅ t4 LLM 返回第二个 token │　　　　　　　　　　　　doOnNext → sink.next() ← Netty 缓冲 t5 LLM 返回第三个 token │　　　　　　　　　　　　doOnNext → sink.next() ← Netty 缓冲 ... 更多 token │　　　　　　　　　　　　　│ tn LLM 流结束 │ ← blockLast 返回！　　　　│ tn+1 执行工具 → sink.next() │ ← 同一个线程 　　　│ tn+2 递归下一轮 │　　　　　　　　　　　　　│

关键问题：t3 时刻的 sink.next() 是 EventLoop 线程上的第一次写入，Netty 会立即 flush。但 t4、t5 的 sink.next() 也来自 EventLoop 线程，Netty 的 write coalescing 会把它们合并——想等更多的数据再一起发。

等到 tn 时刻 blockLast() 返回，boundedElastic 线程苏醒，连发一串 sink.next()（工具调用等），Netty 终于 trigger 一个 flush——前端一次性收到几十个事件。

4.4 Netty Write Coalescing 详解

什么是 write coalescing？

Netty 为了提高网络吞吐，不会每写一个字节就发一个 TCP 包——它会攒一批再发。这是网卡层面常见的优化（Nagle 算法）。

// Netty 的写操作本质上是异步的
channel.write(data);       // 把数据放到发送缓冲区
channel.flush();            // 真正把缓冲区的数据发出去

// 合在一起：
channel.writeAndFlush(data); // 写+立即发送

如果多次调用 write() 但不调 flush()，Netty 会等 EventLoop 的下一个 tick 才 flush。这就是问题所在：

EventLoop 的一个 tick： ┌─────────────────────────────────────────────────┐ │ 处理 IO 事件 → 处理 task queue → 处理 scheduled │ │ ↑ 这里包含 doOnNext → sink.next() │ │ ↑ write coalescing 把多次 write 合并了 │ │ ↑ 只在 tick 结束时 flush 一次 │ └─────────────────────────────────────────────────┘ 前端看到的效果： 第一个 tick → flush → 前端收到第一个 token ✅ 第二个 tick → 写入但不 flush（等更多数据） 第三个 tick → 写入但不 flush ... → ... blockLast 返回 → boundedElastic 线程写入 → flush 📦 → 前端收到全部剩余事件

为什么第一个事件能收到？因为新的 TCP 连接建立后，首次写入通常会触发立即 flush（Netty 的 ChannelOutboundBuffer 在从空到非空时会设置 flush 调度）。

修复后的核心代码：

// 阶段 1：LLM 流式输出
// flatMap 在 EventLoop 上 emit thinking_token → SSE writer 同线程处理
Flux<StreamChunk> thinkingFlux = chatModel.stream(state.getFullMessages(), tools)
    .doOnNext(chatResponse -> {
        // 副作用：累计文本和工具调用
        AssistantMessage msg = ...;
        thoughtBuffer.append(msg.getText());
        toolCalls.addAll(msg.getToolCalls());
    })
    .flatMap(chatResponse -> {
        // 实时推送 thinking_token —— EventLoop 同线程！
        AssistantMessage msg = ...;
        if (msg.getText() != null) {
            return Mono.just(StreamChunk.thinkingToken(
                msg.getText(), conversationId, "迭代" + iterNum));
        }
        return Mono.empty();
    });

// 阶段 2：工具执行
// publishOn(boundedElastic) 确保同步 tool.call() 不阻塞 EventLoop
Flux<StreamChunk> continuationFlux = Flux.defer(() -> {
    // 执行工具...
    // 递归下一轮...
}).publishOn(Schedulers.boundedElastic());

// 组装：先 LLM 流，再工具执行
return thinkingFlux.concatWith(continuationFlux);

为什么修复了：

  1. 同线程 emission：thinkingFlux 通过 flatMap 在 EventLoop 上 emit → SSE writer 在同一 EventLoop 上处理 → 每次写入后框架自动 flush。没有跨线程问题。
  2. 线程职责分离：publishOn(boundedElastic) 明确切分——IO 密集型操作（接收 LLM token）在 EventLoop，CPU/阻塞型操作（同步 tool.call()）在 boundedElastic。
  3. 无 blockLast：不再阻塞任何线程等待 Inner Flux 完成。Inner Flux 通过 concatWith 自然流转到下一阶段。
修复后的线程流向： ┌─ EventLoop 线程 ─────────────────────────────┐ │ chatModel.stream() │ │ ↓ flatMap (同线程 emit) │ │ SSE writer (同线程处理 + 自动 flush) │ │ → 前端逐 token 收到 thinking_token ✅ │ └──────────────────────────────────────────────┘ │ ▼ concatWith │ ┌─ boundedElastic 线程 ────────────────────────┐ │ Flux.defer → 执行 tool.call() │ │ → emit tool_call / tool_result │ │ → 递归下一轮 nextIteration() │ └──────────────────────────────────────────────┘

4.6 教训总结

✅ 教训 1：Flux.create 的 sink.next() 线程安全 ≠ 下游写入线程安全
sink.next() 可以从任何线程调用，但下游的 IO 写入器（Netty）的 flush 行为取决于调用线程。跨线程调用可能导致 write coalescing 产生非预期的缓冲。
✅ 教训 2：纯反应式管道天然避免此问题
flatMap → concatWith 的链式中，emission 和 consumption 在同一线程，Netty 的 flush 语义自然正确。
✅ 教训 3：subscribeOn vs publishOn 选型
subscribeOn 把整个上游移到一个线程池；publishOn 只切分下游。对于"一部分在 EventLoop，一部分在 boundedElastic"的场景，publishOn 更精确。
✅ 教训 4：blockLast() 在 Flux.create 内部是 code smell
blockLast() 在响应式代码内部出现通常意味着设计有问题。它把异步拉回同步，丧失了响应式的大部分优势。
✅ 教训 5：Flux.create + subscribeOn(boundedElastic) 在伪流式阶段能工作
当 LLM 调用是 chatModel.call()（同步阻塞）时，整个 lambda 里只有一个线程从头跑到尾，没有跨线程问题。切换到真流式 chatModel.stream() 后，内部出现两个线程交叉调用 sink.next()，问题才暴露。这不是方案选错了，而是方案随着架构演进而变得不合适。
